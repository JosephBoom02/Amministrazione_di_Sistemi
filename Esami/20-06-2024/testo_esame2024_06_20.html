Amministrazione di Sistemi 6 CFU - Parte 2 - Pratica [20240620]


Contesto dell'esercizio

Si consideri l'infrastruttura in figura con una rete di client "C", un router "R" e una rete di server "S" .

Per semplicità, si svolgano i test attivando due client e un server, ma gli script devono essere progettati in modo da funzionare anche per un numero arbitrario di client e server funzionanti.



Di seguito sono dettagliate le attività da svolgere.


Configurazione e modalità di consegna


Create sulla macchina Linux host una cartella esame e all'interno di quella il Vagrantfile e una gerarchia standard di cartelle e file Ansible per generare le VM, configurarle, e trasferirvi i file necessari.

Per consegnare, create un archivio con nome MATRICOLA-20-06-2024.tar.gz , in cui MATRICOLA sia il vostro numero su 10 cifre, e il tipo di file sia coerente con l'estensione, e inseritevi la cartella esame (e ovviamente tutto ciò che contiene).

Solo questo archivio deve essere consegnato via EOL. La struttura dell'archivio e la correttezza dei nomi saranno valutate.



Rete e servizi

R deve avere le seguenti caratteristiche
    • indirizzo statico 172.20.0.1/20 su NET1
    • indirizzo statico 192.168.1.1/24 su NET2
    • eroga via DHCP ai Client indirizzi appartenenti alla sottorete 172.20.8.0/21
    • eroga via DHCP ai Server indirizzi appartenenti alla sottorete 192.168.1.128/25
    • Il server di test (una volta verificato il suo indirizzo MAC al primo avvio) deve ricevere sempre l'indirizzo 192.168.1.129

Ci deve essere instradamento tra i Client  e i Server

I soli sistemi Client e Server devono essere configurati per il login centralizzato via LDAP. 

Il directory server è in esecuzione sul server 192.168.1.129.

Su tutte le macchine Client e Server il demone di logging deve essere configurato per scrivere i messaggi etichettati local2.warning nel file /var/log/users.log.

Sul Router deve essere installato l'agent SNMP accessibile dai Client.



Creazione utenti


Si realizzi uno script /home/vagrant/new-user.sh collocato su R ed eseguibile unicamente dall'utente vagrant, che accetti come parametro un nome utente.

Lo script deve verificare se l'utente esiste in in LDAP, e in caso contrario, eseguire queste operazioni (nel seguito naturalmente USERNAME rappresenta il nome dell'utente creato):

  • chiede interattivamente una password, ripetendo la richiesta finché non viene inserita una stringa di almeno 6 caratteri

  • inserisce in LDAP l'utente  impostando la shell a /bin/bash. e un gruppo omonimo, curando automaticamente la scelta degli identificatori numerici

  • crea una coppia di chiavi RSA salvandole in /home/vagrant/keys/USERNAME  e  /home/vagrant/keys/USERNAME.pub

  • invia USERNAME e la corrispondente chiave pubblica attraverso syslog, utilizzando l'etichetta local2.warning, a ogni Client e a ogni Server del range di indirizzi teoricamente forniti da DHCP

  • configura l'agent SNMP creando una nuove estensione con nome USERNAME in modo che una richiesta al corrispondente managed object dia come risposta la chiave privata dell'utente USERNAME




Creazione delle home e configurazione del login


Uno script /bin/get-users.sh dovrà essere disponibile su tutti i Client e tutti i Server , eseguibile solo dall'utente root

Lo script deve essere avviato automaticamente al boot e riavviato in caso di terminazione per qualsiasi causa.

Lo script resta in attesa di nuove righe in /var/log/users.log

Per ogni riga, estrae l'utente, crea se necessario la home directory nella forma /home/USERNAME

Per consentire agli utenti dei Client di accedere automaticamente ai Server, lo script procede poi come segue:

  • Se lo script rileva di essere in esecuzione su di uno dei Client, interroga via SNMP il Router per ottenere la chiave privata e la installa nella posizione/nome standard (/home/USERNAME/.ssh/id_rsa) e installa la chiave pubblica prelevata dal log in /home/USERNAME/.ssh/id_rsa.pub

  • Se lo script rileva di essere in esecuzione su di uno dei Server, installa la chiave pubblica tra le chiavi autorizzate all'accesso dell'utente con autenticazione senza password.

  • Si curi la correttaezza di tutti i permessi, mantenendoli il più possibile restrittivi.
